# Phase 1a: VIOトラッキング + デバッグHUD

## このフェーズで入った概念(1つだけ)

**「スマホが自己位置をcm単位で知っている」** — ARCoreのVIO(visual-inertial odometry)。

カメラ映像から床や壁の模様(特徴点)を毎フレーム検出し、「特徴点の見え方がフレーム間でどうズレたか」+加速度・ジャイロの値を融合して、端末が3D空間内でどれだけ動いたかを連続計算する。コンパス・歩数・GPSは一切使っていない。旧Web版の精度問題(磁気の乱れ、歩幅誤差の蓄積)はこの時点で全て消えている。

## 座標系の意味

- 原点(0,0,0) = **ARセッションを開始した瞬間の端末の位置**。アプリを起動し直すと原点もリセットされる(これが後のフェーズ1bでマーカーが必要になる理由)
- 単位はメートル。y軸が上下(重力方向をARCoreが自動検出)、x/z が水平面
- HUDの「ポーズ」はこの座標系での現在位置

## HUDの各値の読み方

| 表示 | 意味 | おかしいときに疑うこと |
|---|---|---|
| 状態 | TRACKING=自己位置を把握中 / PAUSED=見失い中 / STOPPED=セッション停止 | PAUSEDが続く→下の「理由」を見る |
| 理由 | PAUSEDの原因(特徴点不足・暗すぎる・動きが速すぎる等) | 環境の問題かバグかをここで切り分ける |
| ポーズ x,y,z | セッション原点からの現在位置(m) | 静止中に値が流れる→VIOドリフト(通常は±数cm以内) |
| 累積移動距離 | 毎フレームの移動量の合計 | 静止中もノイズでじわじわ増えるのは正常(数分で数十cm程度なら許容) |
| 直線距離 | 原点から現在位置までの直線(m) | 3m歩いて2mしか出ない→トラッキング品質低下を疑う |
| FPS | フレーム更新頻度 | Pixel 3aで30近く出ていれば健全 |

## どういう環境で狂うか(VIOの弱点)

1. **無地の面**(白い壁・天井・単色の床) — 特徴点が取れず PAUSED になる。初回起動時のスクリーンショットで実際に観測済み
2. **暗い部屋** — カメラ映像のコントラストが落ち特徴点検出が不安定になる
3. **速い動き・激しい振り回し** — フレーム間で特徴点を追いきれず一時的にロストする(ロスト中はIMUのみで数秒しのぎ、復帰時に補正が入る)
4. **鏡・ガラス・動くもの(テレビ画面等)** — 「動かない背景」という前提が崩れ、まれに位置が飛ぶ

## 実機での確認手順(Phase 1a 完了条件)

1. アプリ起動 → カメラを部屋の中(模様のあるもの)に向ける → 状態が **TRACKING** になりポーズに数値が入る
2. その場で3mほど歩く → 直線距離が**約3m**になる(誤差±20cm程度なら合格)
3. 元の場所に戻る → 直線距離が**ほぼ0**に戻る(戻らない=ドリフト量が見える)
4. カメラを手で塞ぐ → 状態が **PAUSED** になり、外すと復帰する

## コード構成(概念1対1)

- `ArTrackingManager.kt` — ARCoreセッションの生成/再開/停止とフレーム更新。毎フレーム `TrackingSnapshot`(状態・ポーズ・距離・FPS)を発行する。**VIOに関する責務はこのファイルだけ**
- `CameraBackgroundRenderer.kt` — カメラ映像を画面に描く(OpenGL)。トラッキングには関与しない
- `DebugHud.kt` — TrackingSnapshotを画面上部のテキストに整形するだけ
- `MainActivity.kt` — カメラ権限の要求、ARCore(Google Play開発者サービスAR)のインストール確認、上記3つの接続
