# Phase 1b: マーカー位置合わせ(Augmented Images)

## このフェーズで入った概念(1つだけ)

**「紙マーカーが座標系の原点になる」**

VIOの座標系(Phase 1a)はセッション開始位置が原点のため、アプリを起動し直すたびにリセットされる。物理的に動かないマーカーを認識した瞬間に「マーカーから見た座標」へ変換することで、**起動をまたいで同じ意味を持つ座標系**が手に入る。これが経路の保存(1c)・再利用(1d)の土台。

## 仕組み: なぜ斜めから映しても正しい座標になるか

アプリはマーカーの (1)模様の配置(登録PNG) と (2)物理サイズ(一辺15cm) を知っている。カメラ画像内でマーカーの特徴点が「どの大きさで・どう歪んで」写っているかから、カメラとマーカーの相対位置・角度が一意に逆算できる(PnP問題)。遠い→小さく写る、斜め→台形に潰れる、という歪みがそのまま幾何情報になっている。

処理の流れ:
1. マーカー認識 → カメラから見たマーカーの姿勢をPnPで推定
2. VIO座標系でのカメラ位置は常に既知 → 合成して「マーカーがVIO空間のどこにどの向きであるか」が確定(この点にAnchorを作成)
3. 以後 `markerPose⁻¹ ∘ cameraPose` で「マーカーから見た自機位置」(HUDの mx/my/mz)を毎フレーム計算

認識後はARCoreがAnchor位置をVIOで継続補正するため、マーカーが視界から外れても mx/my/mz は維持される。

## 実機で確認済みの性質(2026-07-21, Pixel 3a)

- マーカー認識で mx/my/mz が確定し、視界から外れても値が維持される
- **タスクキル→再起動→マーカー再認識で、同じ立ち位置の mx/my/mz がほぼ同じ値に復元される**(1bの核心)
- 斜めからの再認識でも正しい座標が出る(上記PnPの説明どおり)
- ホーム画面経由の復帰はプロセスが生きているため、再認識なしで値が残る(こちらはセッション継続でありマーカーの効果ではない)

## ハマった不具合と教訓

**症状**: 初版マーカーで「状態: STOPPED / ARCoreセッションを作成できませんでした」

**原因**: 生成したマーカー画像が特徴点不足で、`AugmentedImageDatabase.addImage` の品質チェックに拒否されていた。`arcoreimg` ツール(品質採点、75点以上推奨)で70点。図形が大きく数が少なすぎた。

**対処**: 大小数十個の非対称図形+多数の角を持つパターンに再生成し100点に。あわせて品質不足の専用エラーメッセージを追加した(総称メッセージ「セッションを作成できませんでした」ではlogcatを見るまで原因が分からなかったため)。

**教訓**: マーカー画像を変更したら必ず `tools/arcoreimg/arcoreimg.exe eval-img --input_image_path=...` で85点以上を確認してからビルドする。

## マーカーの運用

- 印刷: `docs/marker/marker-print.html` をA4・100%スケールで印刷(一辺15cm)。この15cmという値はコード内 `MARKER_PHYSICAL_WIDTH_METERS` と一致している必要がある(距離推定の基準)。サイズを変えて印刷したら定数も変えること
- 貼る場所: 先延ばしの定位置(動画を見るソファ・ベッド)の近くで、誘導開始時にカメラで映しやすい壁や家具
- PC画面表示でもテスト可能だが、画面上の実寸が15cmでない場合は距離表示がその比率でズレる

## コード構成

- `MarkerAnchor.kt` — AugmentedImageDatabase登録、検出、Anchor管理、マーカー座標系への変換。**マーカーに関する責務はこのファイルだけ**
- `tools/GenerateMarker.java` — マーカー画像の生成(乱数シード固定で再現可能)
- `tools/arcoreimg/` — 品質検証ツール(gitignore対象、ARCore SDK 1.54.0より)
